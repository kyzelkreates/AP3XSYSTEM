BUG-002  SEVERITY: MEDIUM
  Title:     UI deploy buttons bypass the Master Controller rule
  Files:     ui/fleet-detail.html (line 224), ui/dashboard.html (line 236)
  Detail:    Both pages contain JavaScript deploy functions that write deployment
             records directly to localStorage (the in-browser store stub) without
             calling /api/deployFleet and without passing initiator: "master_controller".
             This means:
             - The deployment engine, pre-flight checks, versioning, packaging,
               and environment adapters are all bypassed
             - Fleet-detail and dashboard can self-deploy from the browser, which
               violates the "Only Master Controller can deploy" rule

             fleet-detail.html triggerDeploy():
               Creates a raw deployment object and writes it to store.deployments
               directly — no API call, no pre-flight, no bundle

             dashboard.html deployFleet():
               Same pattern — direct store mutation

  Fix:       Replace both functions with fetch('/api/deployFleet', { method: 'POST',
             body: JSON.stringify({ fleetId, initiator: 'master_controller', ... }) })
             The API handler already enforces the initiator check and returns 403
             if the rule is violated.

  Title:     hazard/confirm.js and hazard/dispute.js bypass hazard-manager
  Files:     api/hazard/confirm.js, api/hazard/dispute.js
  Detail:    Both handlers perform inline store mutation instead of calling the
             purpose-built confirmHazard() / disputeHazard() functions in

             The hazard-manager functions contain additional logic:
             - Vote tallying with deduplication per driverId
             - Threshold-based auto-resolve with configurable threshold
             - Correct event payload structure
             - Edge cases (already-resolved hazards, duplicate votes)
             The API handlers reimplement a simplified version of this logic,
             creating a divergence risk.

  Fix:       Import and call:
               import { confirmHazard, disputeHazard } from "../../core/hazards/hazard-manager.js"
             Replace inline mutation with:
               confirmHazard(store, fleetId, hazardId, driverId)
               disputeHazard(store, fleetId, hazardId, driverId)

  DEPLOYMENT PIPELINE STAGES:
    1. Acquire deploy lock (10 min TTL, prevents concurrent deploys)
    2. Pre-flight validation (8 checks: fleet exists, active, vehicles, drivers,
       devices, identity bound, regulation set, no concurrent deploy)
    3. Register version (semver increment, changelog, target)
    4. Pack (full/incremental/config-only/pwa-only based on bundleTarget param)
    5. Validate bundle (manifest, required sections, entity warnings)
    6. Adapt to environment (Vercel API spec / server artifacts / edge bindings)
    7. Execute deploy (API call or queued for handler)
    8. Health check (device heartbeat rate, bundle checksum verification)
    9. Activate version (supersedes previous, emits deployment.complete)

  ROLLBACK STRATEGIES:
    IMMEDIATE:     Hard cut to prior version. Packs rollback bundle, activates,
                   supersedes current. Completes in <1 event loop.
    GRACEFUL:      Marks deployment as pending_rollback. Waits for active
                   driver tacho sessions and routes to close. isGracefulRollbackReady()
                   + completeGracefulRollback() polled by API handler.
    CANARY_ABORT:  Marks current as FAILED (not ROLLED_BACK), then executes
                   immediate restore of prior version.

  BUNDLE TARGETS:
    FULL:          Complete snapshot — all 10 sections including entity data
    INCREMENTAL:   Diff from prior bundle — only added/modified/removed records
    CONFIG_ONLY:   Fleet config + branding — no entity data (fastest)
    PWA_ONLY:      Config + branding + driver stubs — server unchanged
    ROLLBACK:      Copy of prior bundle with new rollback metadata

  INITIATOR ENFORCEMENT:
    orchestrator:  Throws Error if options.initiator !== "master_controller"
    API handler:   Returns HTTP 403 with code INITIATOR_NOT_MASTER_CONTROLLER
